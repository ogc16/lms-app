Introduction to Object-Oriented Analysis & Design

What is Object-Oriented Analysis (OOA)?

Object-Oriented Analysis (OOA) is the process of examining a problem domain from the perspective of objects and their interactions. The focus is on understanding what the system needs to do, identifying the key objects in the problem domain, and defining their attributes and behaviors. OOA is concerned with the "what" rather than the "how."

Key output of OOA: A conceptual model consisting of classes, objects, attributes, and operations that represent the problem domain.

What is Object-Oriented Design (OOD)?

Object-Oriented Design (OOD) is the process of defining how the system will be built using object-oriented concepts. It takes the analysis model and adds implementation details, defining the software objects, their interfaces, and their interactions. OOD is concerned with the "how."

Key output of OOD: A detailed design model that can be directly implemented in an object-oriented programming language.

Benefits of OOAD

  • Reusability — Classes and objects can be reused across projects, reducing development time and cost.
  • Maintainability — Object-oriented systems are easier to modify and extend due to modular design.
  • Better Modeling — The real world is naturally represented through objects, making models more intuitive.
  • Improved Communication — Unified Modeling Language (UML) provides a common vocabulary for stakeholders.
  • Reduced Complexity — Systems are broken down into manageable objects and classes.
  • Faster Development — Iterative development and reuse accelerate the development process.

Object-Oriented Concepts & Principles

The Four Pillars of Object-Orientation

1. Encapsulation

Encapsulation is the bundling of data (attributes) and methods (operations) that operate on that data into a single unit (a class), while restricting access to some of the objects internal components. This protects the integrity of the data and prevents unintended interference.

Example: A BankAccount class encapsulates the balance attribute and provides public methods like deposit() and withdraw() to interact with it, preventing direct modification of the balance.

2. Inheritance

Inheritance is a mechanism where a new class (child/subclass) derives properties and behaviors from an existing class (parent/superclass). It promotes code reuse and establishes a hierarchical relationship between classes.

Example: SavingsAccount and CheckingAccount inherit from BankAccount, reusing common attributes and methods while adding specialized features.

3. Polymorphism

Polymorphism ("many forms") allows objects of different types to respond to the same message (method call) in different ways. It enables code to work with objects of multiple types through a common interface.

Example: A method calculateInterest() can behave differently for SavingsAccount and LoanAccount objects, even though both are treated as Account references.

4. Abstraction

Abstraction is the process of hiding complex implementation details and showing only the essential features of an object. It simplifies the view of an object by focusing on what it does rather than how it does it.

Example: A Vehicle abstract class defines start() and stop() methods without specifying how they work; concrete subclasses like Car and Motorcycle provide implementations.

Relationships in Object-Oriented Systems

  • Association — A general relationship where one class uses or knows about another. E.g., a Teacher teaches Student.
  • Aggregation — A "has-a" relationship where the part can exist independently of the whole. E.g., a Department has Employee objects. If the department is dissolved, employees still exist.
  • Composition — A strong "has-a" relationship where the part cannot exist without the whole. E.g., a House has Room objects. If the house is destroyed, the rooms cease to exist.
  • Inheritance (Generalization) — An "is-a" relationship where a child class inherits from a parent class. E.g., a Dog is an Animal.
  • Dependency — A temporary relationship where one class depends on another, usually as a method parameter. E.g., a Report class depends on a Printer class to print.

UML Structural Diagrams

What Are Structural Diagrams?

Structural diagrams in UML represent the static structure of a system. They show the elements of the system, their relationships, and their internal structure. Structural diagrams remain constant over time and focus on the architecture of the system.

1. Class Diagram

The class diagram is the most commonly used UML diagram. It shows classes, their attributes, operations, and the relationships among classes. It provides a static view of the systems building blocks.

// Example: Class Diagram Representation +-------------------------+ | Student | +-------------------------+ | - studentID: String | | - name: String | | - email: String | +-------------------------+ | + register(): void | | + enrollCourse(): void | | + getDetails(): String | +-------------------------+ | | (inherits) v +-------------------------+ | GraduateStudent | +-------------------------+ | - thesisTitle: String | +-------------------------+ | + submitThesis(): void | +-------------------------+

2. Object Diagram

An object diagram shows instances of classes (objects) at a specific point in time. It is a snapshot of the systems state, useful for testing and understanding complex interactions.

While a class diagram shows Student, an object diagram shows alice: Student with specific values like name = "Alice", ID = "S123".

3. Component Diagram

A component diagram shows the physical components of a system, their dependencies, and their interactions. Components represent modular parts of the system that encapsulate implementation.

4. Deployment Diagram

A deployment diagram shows the physical deployment of artifacts on hardware nodes. It models the topology of the systems hardware and the software components running on each node.

5. Package Diagram

A package diagram organizes related elements into groups (packages). It shows dependencies between packages and provides a high-level view of the systems structure.

6. Composite Structure Diagram

A composite structure diagram shows the internal structure of a class, including its parts and their connections. It is used to model complex classes that contain other components.

UML Behavioral Diagrams

What Are Behavioral Diagrams?

Behavioral diagrams in UML represent the dynamic behavior of a system. They show how the system responds to events, how objects interact over time, and the flow of control or data through the system.

1. Use Case Diagram

A use case diagram shows the functionality of a system from the users perspective. It displays actors (external entities) and use cases (system functions) with their relationships.

2. Sequence Diagram

A sequence diagram shows object interactions arranged in time sequence. It emphasizes the order of messages exchanged between objects, making it clear which interactions occur first.

3. Communication Diagram

A communication diagram (formerly collaboration diagram) shows interactions between objects with a focus on the structural organization of the objects. Messages are numbered to show sequence.

4. State Machine Diagram

A state machine diagram shows the life cycle of an object: the states it can be in, transitions between states, and events that trigger transitions. It is useful for modeling event-driven systems.

5. Activity Diagram

An activity diagram shows the flow of control from one activity to another. It resembles a flowchart and is used to model business processes, workflows, and the logic of operations.

6. Timing Diagram

A timing diagram shows the behavior of objects over a specific time period, focusing on timing constraints and state changes. It is used for real-time and embedded systems.

System Development Methodologies

Why a Methodology?

A system development methodology provides a structured, disciplined approach to building information systems. It defines processes, deliverables, roles, and standards that guide the development team through each phase of the project.

  • Consistency — Ensures all projects follow the same proven patterns and practices.
  • Quality — Standardized reviews, testing, and documentation improve quality.
  • Communication — Provides a common language and framework for all stakeholders.
  • Risk Management — Identifies and mitigates risks early in the development process.
  • Predictability — Better estimates of time, cost, and resources.
  • Traceability — Requirements can be traced through design, implementation, and testing.

Agile Methodology

Agile is an iterative, incremental approach to software development that emphasizes flexibility, collaboration, and customer feedback. The Agile Manifesto (2001) values:

  • Individuals and interactions over processes and tools
  • Working software over comprehensive documentation
  • Customer collaboration over contract negotiation
  • Responding to change over following a plan

Popular Agile frameworks include Scrum, Extreme Programming (XP), and Kanban.

How Agile Supports OOAD

Agile and OOAD complement each other in several ways:

  • Iterative Modeling — Agile iterations allow OOAD models to evolve gradually rather than being fully defined upfront.
  • User Stories — Agile user stories align with use case modeling in OOAD.
  • Refactoring — Agile encourages continuous refactoring of both code and design models.
  • Test-Driven Development — TDD supports verifying object interactions during design.
  • Collaboration — Both Agile and OOAD emphasize close collaboration between developers and stakeholders.
In an Agile-OOAD project, models are created "just in time" and "just enough" to guide implementation, rather than being comprehensive documents created upfront.

Requirements Elicitation & Use Case Modeling

Requirements Elicitation Techniques

Requirements elicitation is the process of gathering information from stakeholders to determine what the system must do. Common techniques include:

  • Interviews — One-on-one conversations with stakeholders to understand their needs and expectations.
  • Questionnaires — Written surveys distributed to a large number of stakeholders to collect broad input.
  • Observation — Watching users perform their work to understand current processes and pain points.
  • Document Analysis — Studying existing documents, reports, and systems to extract requirements.
  • Brainstorming — Group sessions where stakeholders generate ideas in a creative, open environment.
  • Prototyping — Building a working model of the system to help stakeholders visualize requirements.
  • Joint Application Development (JAD) — Structured workshops where stakeholders and developers collaborate intensively.
  • Focus Groups — Small groups of stakeholders discussing specific aspects of the system.

Use Case Model Benefits

A use case model captures the functional requirements of a system by describing interactions between actors and the system. Benefits include:

  • User-Centric — Focuses on what users need the system to do.
  • Clear Communication — Provides a simple, intuitive notation understood by all stakeholders.
  • Comprehensive — Captures all functional requirements in a structured way.
  • Traceability — Each use case can be traced through design and testing.
  • Basis for Testing — Use cases form the foundation for acceptance testing.
  • Scope Control — Helps define and manage the boundaries of the system.
  • Estimating — Use cases can be used to estimate development effort and complexity.

Use Case Diagrams & Written Use Cases

Elements of Use Case Diagrams

  • Actor — An external entity that interacts with the system (a person, another system, or hardware). Represented as a stick figure.
  • Use Case — A specific function or behavior of the system. Represented as an oval/ellipse containing the name.
  • System Boundary — A rectangle that encloses all use cases, defining the scope of the system.
  • Association — A line connecting an actor to a use case, indicating communication.
  • <<include>> — A relationship where one use case always includes the behavior of another.
  • <<extend>> — A relationship where a use case optionally extends the behavior of another.
  • Generalization — Shows inheritance between actors or between use cases.

Written Use Case Format

A written use case describes the step-by-step interaction between an actor and the system. A typical template includes:

  • Use Case Name: A unique, verb-noun phrase
  • Actor(s): Who initiates and participates
  • Precondition: What must be true before the use case starts
  • Postcondition: What must be true after the use case completes
  • Main Flow: The normal sequence of steps
  • Alternative Flows: Variations or error conditions
  • Exceptions: Error handling steps

Class Diagrams — Three Views

The Three Views of Class Diagrams

In OOAD, class diagrams can be created at three different levels of abstraction, each serving a distinct purpose.

1. Conceptual View

The conceptual view focuses on the real-world domain concepts and their relationships. It contains no software-specific details and is used to communicate with domain experts and stakeholders.

  • Classes represent real-world concepts (e.g., Patient, Ambulance, Hospital)
  • Attributes are simple (e.g., patient name, ambulance ID)
  • No method signatures or visibility modifiers
  • Focus on understanding the problem domain

2. Specification View

The specification view focuses on the interfaces of software classes. It defines what the classes do without specifying how they do it. This view is useful for designing contracts between components.

  • Classes have interfaces (methods) defined
  • Attributes still high-level
  • Focus on what the class provides
  • Suitable for designing system architecture

3. Implementation View

The implementation view is the most detailed view, showing the actual code-level structure of classes. It includes all method implementations, visibility, types, and platform-specific details.

  • Full method signatures with return types
  • Visibility modifiers (+, -, #)
  • Data types for all attributes
  • Constructors and destructors
  • Platform-specific details

Interaction Diagrams

Sequence Diagrams

A sequence diagram is a type of interaction diagram that shows how objects communicate with each other over time. It emphasizes the chronological order of messages exchanged between objects.

Key elements include:

  • Lifeline — A vertical dashed line representing the existence of an object over time.
  • Activation Bar — A thin rectangle on a lifeline showing when an object is executing.
  • Message — An arrow from one lifeline to another representing communication.
  • Self-Message — A message an object sends to itself.
  • Combined Fragment — A rectangle enclosing interaction parts with a guard condition. Types: alt (alternatives), opt (optional), loop (iteration), par (parallel).

A communication diagram (formerly collaboration diagram) shows the same interactions as a sequence diagram but emphasizes the structural organization of objects. Messages are numbered to indicate sequence.

Design Patterns & Case Studies

Three Categories of Design Patterns

Design patterns are reusable solutions to common software design problems. They are categorized into three groups:

1. Creational Patterns

Creational patterns deal with object creation mechanisms, trying to create objects in a manner suitable to the situation.

  • Singleton — Ensures a class has only one instance and provides a global point of access.
  • Factory Method — Defines an interface for creating an object but lets subclasses decide which class to instantiate.
  • Abstract Factory — Provides an interface for creating families of related objects without specifying their concrete classes.
  • Builder — Separates the construction of a complex object from its representation.
  • Prototype — Creates new objects by copying an existing object (prototype).

2. Structural Patterns

Structural patterns deal with object composition, defining how classes and objects can be combined to form larger structures.

  • Adapter — Allows incompatible interfaces to work together.
  • Decorator — Adds responsibilities to objects dynamically.
  • Facade — Provides a simplified interface to a complex subsystem.
  • Composite — Composes objects into tree structures to represent part-whole hierarchies.
  • Proxy — Provides a surrogate or placeholder for another object.

3. Behavioral Patterns

Behavioral patterns deal with object interactions and responsibility distribution.

  • Observer — Defines a one-to-many dependency so that when one object changes state, dependents are notified.
  • Strategy — Defines a family of algorithms, encapsulates each one, and makes them interchangeable.
  • Template Method — Defines the skeleton of an algorithm, letting subclasses redefine certain steps.
  • Command — Encapsulates a request as an object, allowing parameterization and queuing.
  • State — Allows an object to alter its behavior when its internal state changes.

Template Method Pattern

The Template Method pattern defines the skeleton of an algorithm in an operation, deferring some steps to subclasses. It lets subclasses redefine certain steps without changing the algorithms structure.

// Template Method Pattern Example abstract class DataProcessor { // Template method - defines the algorithm skeleton public final void process() { readData(); processData(); writeOutput(); } protected abstract void readData(); protected abstract void processData(); protected void writeOutput() { // Default implementation } }

Why: Striving for low coupling and high cohesion is a fundamental OOAD principle. It directly impacts maintainability, reusability, and testability. Every design decision should consider these two metrics.

Show Answer
LSP: Objects of a superclass should be replaceable with objects of a subclass without affecting correctness.
Violation Example: Class Rectangle has setWidth() and setHeight(). Square extends Rectangle but overrides setWidth() to also set height, and setHeight() to also set width. A client calling rect.setWidth(5); rect.setHeight(10) expects width=5, height=10 but with a Square, gets width=10, height=10 — breaking the contract.

Why LSP matters: Without LSP, polymorphism introduces subtle bugs. Code that works with a base class may fail when a subclass is substituted, defeating the purpose of polymorphism.
